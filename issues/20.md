# Issue 20 — Artist index: background cache refresh

## Status

Future improvement (not blocking).

## Background

`BuildArtistIndex` scans all albums to build the artist index and caches the result for 15 minutes.
Currently, stale-cache hits (every 15 min) block the request thread while the index is rebuilt.

The original design intent (noted in CLAUDE.md) was stale-while-revalidate: return stale data
immediately, rebuild in background. The `async` keyword was present in the original code as a
placeholder for this, but no `Task.Run` or background work was ever wired up. The method was
made synchronous during the Issue 8 cleanup pass to reflect what the code actually does.

## Proposed implementation

On a stale-cache hit:
1. Return the stale index immediately (fast path for the current request)
2. Fire a background `Task.Run` to call `BuildArtistList` and update `derived_cache`
3. Guard with a static per-cache-key lock (e.g. `HashSet<string>` + `lock`) to prevent
   concurrent parallel rebuilds if multiple requests arrive simultaneously while stale

Cold cache (no entry at all) still blocks synchronously — unavoidable on first request.

## Considerations

- `User` entity and `folderIds` list captured into the background task need to be safe to use
  off the request thread (they're read-only value-type lookups, should be fine)
- SQLite write in `SetDerivedCache` from a background thread is safe as long as `SubsonicStore`
  uses WAL mode or serializes writes (verify before shipping)
- Performance today is acceptable for typical library sizes; this is only valuable for very large
  libraries where `BuildArtistList` takes >100ms
