# getAlbumList?type=recent returns wrong albums

**status:** closed — confirmed working in v0.2.1.0

## Symptom

In Tempus (and likely all Subsonic clients), "Recently Played" albums returns albums that don't match what was actually listened to recently. Example discrepancy:

**What Subsonic returns (wrong):**
1. Smoking Monkey — Jonathan Coulton
2. They Sure Don't Make Basketball Shorts Like They Used To — Hoobastank
3. Steppin' Out: Astaire Sings

**What Jellyfin's UI shows as recently played tracks (correct):**
1. Money for Nothing — Dire Straits
2. I Ain't the One — Lynyrd Skynyrd
3. Invisible Touch — Genesis

## Root cause

`getAlbumList?type=recent` queries `MusicAlbum` items sorted by `ItemSortBy.DatePlayed` descending (`SubsonicController.cs` ~line 524). However, Jellyfin only sets `DatePlayed`/`LastPlayedDate` on individual `Audio` items when a track is scrobbled — **not** on the parent `MusicAlbum` entity.

So albums returned are ones with stale or incidentally-set album-level play dates, not "albums whose songs were recently played." The actual scrobble data lives on `Audio` items.

## Fix options

**Option A (recommended): propagate play date to album in scrobble handler**

In the `scrobble` endpoint (after saving user data on the `Audio` item), look up the parent album and update its `LastPlayedDate` too:

```csharp
// After saving play data on the song:
var album = _library.GetItemById(song.ParentId) as MusicAlbum;
if (album != null)
{
    var albumData = _userData.GetUserData(user, album);
    // Only update if this play is more recent than the album's current LastPlayedDate
    if (albumData.LastPlayedDate == null || albumData.LastPlayedDate < data.LastPlayedDate)
    {
        albumData.LastPlayedDate = data.LastPlayedDate;
        _userData.SaveUserData(user, album, albumData, UserDataSaveReason.PlaybackFinished, CancellationToken.None);
    }
}
```

This keeps the `getAlbumList` query unchanged; the album's `DatePlayed` just becomes accurate.

**Option B: derive recency from songs at query time**

Query `Audio` items sorted by `DatePlayed` descending, deduplicate by `ParentId` (album), then fetch those albums. More expensive but doesn't require scrobble-time changes.

## Notes

- Option A is simpler and fits the existing pattern (scrobble already calls `SaveUserData` synchronously).
- The stale data already in the DB won't self-correct — users would need to play albums again for them to surface accurately. That's acceptable.
- Same bug likely affects `getAlbumList2?type=recent` (same handler, `v2=true`).
- `type=frequent` (sort by play count) may have a similar problem if play counts aren't propagated to the album level, but hasn't been reported yet.
